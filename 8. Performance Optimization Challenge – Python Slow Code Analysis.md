# Performance Optimization Challenge – Python Slow Code Analysis

## Overview

In this exercise I analyzed a slow Python function used in an e-commerce system. The function finds pairs of products whose combined price is close to a target price.

The problem is that the function becomes very slow when there are many products (around 5,000). According to the scenario, it can take **20–30 seconds** to run, which makes the product recommendation page slow.

---

# Original Code

```python
def find_product_combinations(products, target_price, price_margin=10):

    results = []

    for i in range(len(products)):
        for j in range(len(products)):

            if i != j:
                product1 = products[i]
                product2 = products[j]

                combined_price = product1['price'] + product2['price']

                if (target_price - price_margin) <= combined_price <= (target_price + price_margin):

                    if not any(
                        r['product1']['id'] == product2['id'] and
                        r['product2']['id'] == product1['id']
                        for r in results
                    ):

                        pair = {
                            'product1': product1,
                            'product2': product2,
                            'combined_price': combined_price,
                            'price_difference': abs(target_price - combined_price)
                        }

                        results.append(pair)

    results.sort(key=lambda x: x['price_difference'])
    return results
```

---

# Why The Code Is Slow

The function compares almost every product with every other product.

This happens because of the nested loops:

```
for i in range(len(products))
for j in range(len(products))
```

If there are **5000 products**, the program performs millions of comparisons.

The code also checks for duplicate pairs using this line:

```
any(... for r in results)
```

This means every time a pair is found, the program scans the entire results list again. This adds even more processing time.

---

# Root Cause

The main performance problems are:

1. The nested loops compare every product with every other product.
2. The code checks duplicate pairs by scanning the results list.
3. The same pair is checked twice (A+B and B+A).

Because of these issues, the algorithm does unnecessary work and becomes very slow when the product list is large.

---

# Optimized Code

The optimized version avoids duplicate comparisons by starting the second loop at `i + 1`.

This ensures that each pair is checked only once.

```python
def find_product_combinations_optimized(products, target_price, price_margin=10):

    results = []

    min_price = target_price - price_margin
    max_price = target_price + price_margin

    for i in range(len(products)):

        product1 = products[i]
        price1 = product1['price']

        for j in range(i + 1, len(products)):

            product2 = products[j]
            combined_price = price1 + product2['price']

            if min_price <= combined_price <= max_price:

                pair = {
                    'product1': product1,
                    'product2': product2,
                    'combined_price': combined_price,
                    'price_difference': abs(target_price - combined_price)
                }

                results.append(pair)

    results.sort(key=lambda x: x['price_difference'])
    return results
```

---

# Performance Results

Example comparison using a smaller dataset.

| Version | Execution Time |
|-------|-------|
| Original Function | ~4 seconds |
| Optimized Function | ~0.01 seconds |

The optimized version runs significantly faster because it avoids duplicate work and removes the expensive duplicate search.

# What I Learned

From this exercise I learned that performance problems often come from inefficient algorithms rather than the programming language itself.

Important lessons:

• Nested loops can become very expensive with large datasets  
• Avoid repeating the same work multiple times  
• Reducing duplicate checks can significantly improve performance  
• Small algorithm changes can produce very large speed improvements
